# GitLab CI Security Assurance Pipeline

## Purpose

This repository now includes a GitLab-native CI pipeline that treats build output as part of the evidence chain for Vigilant ISSO itself. The goal is not only to catch syntax or security issues, but to preserve a repeatable record showing what was scanned, when, and with which tooling.

That aligns with the broader Vigilant ISSO discipline:
- **NIST-oriented evidence quality** - retain source artifacts, timestamps, and scan outputs that can be reviewed later.
- **AWS Well-Architected Security** - detect misconfiguration, least-privilege drift, and insecure build inputs early.
- **AWS Well-Architected Reliability and Operational Excellence** - make validation repeatable, deterministic, and visible in every merge path.
- **AWS Well-Architected Cost Optimization** - skip tool-specific jobs cleanly when the repository has no relevant Terraform, Python, or Docker inputs instead of burning runner time on empty scans.

## Pipeline stages

1. **inventory**
   - Detects whether Terraform, Python, or Docker-relevant files exist.
   - Publishes a dotenv profile used by later jobs.
   - Produces a repo inventory artifact so reviewers can see exactly what the pipeline believed it scanned.

2. **validate**
   - Runs `terraform fmt -check -diff -recursive` and per-directory `terraform validate` when Terraform exists.
   - Writes explicit skip evidence if no Terraform is present.

3. **security**
   - **tfsec** for Terraform-specific static analysis when Terraform exists.
   - **checkov** for repository-wide IaC and configuration scanning.
   - **trivy fs** for vulnerability, misconfiguration, and secret scanning.
   - **bandit** for Python-focused security analysis when Python exists.

4. **sbom**
   - Generates a CycloneDX SBOM using Trivy so the repo has a machine-readable component inventory artifact.

5. **report**
   - Publishes a simple index of the generated artifacts so operators and reviewers can find the relevant evidence quickly.

## Artifact expectations

The pipeline intentionally keeps machine outputs as artifacts instead of collapsing everything into a pass/fail light.

Expected artifacts include:
- `reports/repo-inventory.txt`
- `reports/repo-profile.env`
- `reports/terraform-fmt.txt`
- `reports/terraform-validate.txt`
- `reports/tfsec.json`
- `reports/checkov.json`
- `reports/checkov.sarif`
- `reports/trivy-fs.json`
- `reports/trivy-misconfig.sarif`
- `reports/bandit.json`
- `reports/repository-sbom.cdx.json`
- `reports/security-report-index.md`

These should be treated as **review inputs and evidence artifacts**, not as automatic authorization to deploy, close POA&Ms, or claim readiness.

## Security Vulnerability Reporting (SVR)

For this repository, SVR should be handled as a governed process:

1. Use Trivy, Checkov, tfsec, and Bandit outputs as machine-generated intake.
2. Normalize material findings into a human-tracked issue, POA&M, or remediation record.
3. Capture:
   - source tool and artifact path
   - commit SHA / pipeline ID
   - affected file or component
   - severity and confidence
   - remediation owner
   - due date / target milestone
   - validation evidence after closure
4. Preserve the original artifact even after a finding is remediated.
5. Require closure validation, not just code change claims.

This is consistent with Vigilant ISSO’s broader model: **findings become governed work items only after scope, severity, and evidence quality are reviewed by humans.**

## SBOM guidance

The CycloneDX SBOM generated by the pipeline is useful for:
- documenting repository contents at a point in time
- supporting downstream software supply chain review
- correlating later vulnerability advisories to the exact pipeline run that produced the artifact

Recommended handling:
- retain SBOM artifacts with the same retention window as other review evidence
- tie SBOM review to release or monthly readiness checkpoints
- treat unusual or empty SBOM output as a review signal, not as something to ignore silently

## Dockerfile note

At the time this pipeline was added, the repository did **not** contain any Dockerfiles. The pipeline therefore does not force a container-image build job. If Dockerfiles are introduced later, the repo inventory will expose that change and the pipeline should be extended with image build, image scan, and Dockerfile lint controls.
